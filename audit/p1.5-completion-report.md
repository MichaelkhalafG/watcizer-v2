### P1.5-2 — GuestCartMiddleware
**What was changed:** New `backend/app/Http/Middleware/GuestCartMiddleware.php` — resolves identity from JWT (`sub` → `['user_id']`) or `X-Guest-Token` header (minting a UUID when absent → `['guest_token']`), merges it onto the request, and echoes the guest token on the response. Registered alias `guest.cart` in `app/Http/Kernel.php` (`$middlewareAliases`). Applied `guest.cart` to `add_to_cart`, `delete_cart/{id}`, `add_wishlist`, `delete_wishlist/{id}`, `add_order`, `me/cart` in `routes/api.php`.
**Verified by:** curl POST `/api/add_to_cart` with no token → response carried a freshly minted `X-Guest-Token`; with a supplied token → response echoed the same token; php -l clean.

### P1.5-3 — resolveCart + cart endpoints
**What was changed:** `backend/app/Http/Controllers/Api/OrderController.php` — added private `resolveCart(array $identity)` (`firstOrCreate` with 7-day `expires_at` for guests, null for users); `AddToCart` now uses `$this->resolveCart($request->identity)` and dropped the `user_id` validation rule; `ShowCart(Request $request)` now looks up `Cart::where($request->identity)`. Approved out-of-file changes: added `guest_token`/`expires_at` to `Cart::$fillable` (`app/Models/Cart.php`) and moved `me/cart` out of the `auth:api` group into a `guest.cart` group (`routes/api.php`).
**Verified by:** curl guest POST `/api/add_to_cart` → 200 (DB row: `user_id` NULL, correct `guest_token`, `expires_at` +7 days); curl guest GET `/api/me/cart` (same token) → 200 returning the cart with the added item; php -l clean.

### P1.5-4 — Frontend guest token + interceptor
**What was changed:** `Frontend/src/Context/api.jsx` — request interceptor injects `Authorization: Bearer` when a JWT exists, else mints/sends `X-Guest-Token` (persisted in `localStorage.wz_guest_token`); new response interceptor persists any server-echoed `X-Guest-Token`. `Frontend/src/Store/cartStore.js` — `addItem` now fires an additive, fire-and-forget `http.post('add_to_cart', …)` after the optimistic local write. Approved out-of-file change: `AddToCart` price validation `integer` → `numeric` (`OrderController.php`) so decimal cart prices persist.
**Verified by:** simulated sync with decimal price `100.5` → 200, persisted as `100.50`; `me/cart` returned the item by guest token; vite build clean; php -l clean.

### P1.5-5 — MergeGuestCart on login/register
**What was changed:** New `backend/app/Services/MergeGuestCart.php` — within a DB transaction, folds a guest cart's items into the user's cart (`updateOrCreate` with `DB::raw("quantity + N")`), then deletes the guest cart. `backend/app/Http/Controllers/Api/AuthController.php` — `login()` calls the merge after JWT issuance when `X-Guest-Token` is present; this also covers `register()`, which delegates to `login()` (avoids a redundant double-merge).
**Verified by:** temp script — guest cart with 2 items merged into the user cart with correct quantities/prices/colors, and the guest cart row was deleted; php -l clean on MergeGuestCart.php.

### P1.5-6 — Cleanup job (PruneGuestCarts)
**What was changed:** New `backend/app/Console/Commands/PruneGuestCarts.php` (`carts:prune`) — deletes expired guest carts (and their items), reports the count. Scheduled `->daily()` in `backend/app/Console/Kernel.php`.
**Verified by:** `php artisan carts:prune` → "Pruned 0 expired guest carts."; `php artisan schedule:list` showed `carts:prune` daily (`0 0 * * *`); php -l clean.

### P1.5-7 — Guest checkout UI
**What was changed:** `Frontend/src/Pages/Checkout/Checkout.jsx` — added `isGuest = !sessionStorage.getItem('token')` and guest `guest_name`/`guest_phone`/`guest_email` fields to `formData`; guests see name/phone/email inputs + a "Login to save your cart & track orders" link instead of the address selector; CTA reads "Checkout as Guest" (guest) / "Place Order" (logged-in); `addOrder` payload includes the guest fields when `isGuest`. Header cart badge required no change — it already renders `productsCount`, which `MyProvider` derives from `useCart()` (works for guests and users alike).
**Verified by:** guest sees the three fields + login link; logged-in path (address selector) unchanged; CTA copy switches on `isGuest`; badge correct for both states; vite build clean.

## Deferred Tasks
None. One **non-blocking follow-up** surfaced in P1.5-7: a guest can fill the contact fields and click "Checkout as Guest," but `addOrder` still requires a `selectedAddress` and the backend `AddOrder` requires `address_id`, while the guest UI shows no address capture. Completing an end-to-end guest order therefore needs guest shipping-address capture (changing order-submission logic / backend `address_id` handling), which was explicitly out of scope for P1.5-7. Recommended as a dedicated follow-up before guest checkout is advertised.
